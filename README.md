# Еволюція відмовостійкої архітектури інформаційної системи

Практична частина дипломної роботи. Проєкт демонструє **поетапну еволюцію**
розподіленої системи обробки замовлень: від наївної синхронної архітектури,
яка руйнується під навантаженням, до відмовостійкої подієво-орієнтованої
системи. Кожен етап усуває конкретний дефект попереднього, що підтверджується
навантажувальними тестами.

> Матеріали цього README безпосередньо відповідають підрозділам **Розділу 3**
> пояснювальної записки. Таблиці результатів і скріншоти, отримані під час
> тестів, переносяться у відповідні підрозділи.

---

## Доменна область

Дві мікрослужби:

| Сервіс | Роль | Порт |
|---|---|---|
| **order-service** | Точка входу. Приймає замовлення, ініціює оплату, зберігає результат. | 8080 |
| **payment-service** | Обробляє платіж. Імітує затримку зовнішнього провайдера (200 мс). | 8081 |

Кожен сервіс має **власну базу даних** (`orderdb` / `paymentdb`) в одному
екземплярі PostgreSQL — слабка зв'язаність на рівні даних.

## Технологічний стек

- **Java 25 (LTS)**, **Spring Boot 4.1**
- PostgreSQL 17, HikariCP
- Resilience4j (Bulkhead, Circuit Breaker, Timeout) — Розділ 3.2
- Apache Kafka, Debezium (CDC) — Розділи 3.3–3.4
- Micrometer + Zipkin + Prometheus + Grafana — Розділ 3.4
- **K6** — навантажувальне тестування (HTML/JSON-звіти)
- Docker Compose — локальне середовище з обмеженими ресурсами

## Середовище тестування (імітація продакшну)

Усі сервіси запускаються в Docker з **навмисно обмеженими ресурсами**, щоб:
1. відтворити умови реального продакшн-розгортання;
2. гарантувати **відтворюваність** результатів між запусками;
3. зробити вузькі місця архітектури спостережуваними вже за помірного навантаження.

| Контейнер | CPU | RAM |
|---|---|---|
| postgres | 1.0 | 1 GB |
| order-service | 2.0 | 1 GB |
| payment-service | 2.0 | 1 GB |

### Запуск

```bash
# Підняти все середовище (збірка образів за першого запуску)
docker compose -f infra/docker-compose.yml up --build

# Зупинити та прибрати дані
docker compose -f infra/docker-compose.yml down -v
```

Перевірка вручну:

```bash
curl -X POST http://localhost:18080/orders \
  -H "Content-Type: application/json" \
  -d '{"customer":"Ivan","amount":100.0}'
```

### Запуск навантажувального тесту

K6 ганяється контейнером на мережі стенду (б'є у внутрішнє ім'я `order-service:8080`):

```bash
docker run --rm --network diploma_default \
  -v "$PWD/load-tests:/scripts" -w /scripts \
  -e BASE_URL=http://order-service:8080 -e PEAK_RPS=1000 \
  -e REPORT=3.2.1-virtual-threads \
  grafana/k6 run baseline.js
```

`-e REPORT=<ім'я>` зберігає звіт у `load-tests/reports/<ім'я>.html` (+ `.json`) —
по одному на фазу, щоб не перезаписувати. Exit code **99 = пороги SLO порушено**
(очікувано на перевантаженні). HTML містить графіки й розбивку результатів для
скріншотів у диплом.

### Спостережуваність та ін'єкція відмов

- **Grafana:** `http://localhost:3000` (анонімний вхід) — дашборд «Circuit Breaker
  — Outage & Recovery Timeline» (потоки за типом результату, латентність, % помилок).
- **Prometheus:** `http://localhost:9090`. k6 шле метрики через remote-write:
  додай до запуску `-o experimental-prometheus-rw` і
  `-e K6_PROMETHEUS_RW_SERVER_URL=http://prometheus:9090/api/v1/write`.
- **Fault injection (chaos):** миттєва зміна затримки payment без рестарту —
  `curl -X POST "http://localhost:8081/admin/delay?ms=2000"` (повернути: `ms=200`).
- **Перемикач resilience:** `PAYMENT_RESILIENCE_ENABLED=false docker compose ... up -d order-service`
  вимикає Timeout+CircuitBreaker для контрольного порівняння «без CB».

## Методика дослідження

Для коректного порівняння етапів усі тести виконуються за єдиною методикою:

- **Модель навантаження:** open model (`ramping-arrival-rate` у K6) — фіксований
  потік запитів за секунду незалежно від часу відповіді сервера. Саме ця модель
  виявляє каскадну відмову, на відміну від closed model.
- **Профіль:** прогрів 50→100 RPS (30 с) → вихід на пік (30 с) → утримання піку
  60 с → спад.
- **Метрики:** пропускна здатність (RPS), **перцентилі** часу відповіді
  (p50/p95/p99), частка помилок (`http_req_failed`). Середнє значення не
  використовується як показник — воно приховує хвости розподілу.
- **SLO (цільовий рівень):** `p99 < 1 c`, частка помилок `< 1%`.
- Кожен етап оцінюється таблицею «до/після» за однакового навантаження.

### Умови експерименту (контрольовані змінні)

Щоб приріст продуктивності в кожній фазі був **наслідком архітектури, а не
послаблення лімітів**, стенд заморожено. Наведені параметри **однакові в усіх
чотирьох підрозділах** і не змінюються:

| Параметр (FIXED) | Значення |
|---|---|
| CPU / RAM: order, payment | 2 ядра / 1 GB кожен |
| CPU / RAM: postgres | 1 ядро / 1 GB |
| Пул потоків Tomcat (платформні) | **50** |
| Черга Tomcat (`accept-count` / `max-connections`) | 100 / 150 |
| Пул HikariCP: order / payment | 10 / 20 |
| `synchronous_commit` (postgres) | `off` |
| Затримка payment (імітація провайдера) | 200 мс |
| Профіль навантаження / SLO | єдині для всіх фаз |

**Незалежна змінна — рівно одна на фазу** (саме її ефект ми вимірюємо):

| Фаза | Що змінюється |
|---|---|
| 3.1 | — (точка відліку) |
| 3.2 | модель потоків: платформні → **віртуальні** |
| 3.3 | патерни стійкості: **Bulkhead / Circuit Breaker / Timeout** |
| 3.4 | модель зв'язку: sync → **async → Kafka** |

> Важливо: віртуальні потоки (3.2) — це зміна **моделі конкурентності**, а не
> «підняття ліміту 50». Вони прибирають саму концепцію фіксованого пулу носійних
> потоків; решта стенду (пули БД, postgres, затримка) лишається тією самою.

# Розділ 3. Програмна реалізація відмовостійкої архітектури

## 3.1. Базова синхронна архітектура та методика навантажувального дослідження

### Архітектура

`Клієнт → order-service → (HTTP) → payment-service → PostgreSQL`

Обробка замовлення (`OrderService.createOrder`) виконується **синхронно**:

1. замовлення зберігається у статусі `PENDING` (коротка транзакція);
2. потік **блокується** на HTTP-виклику `payment-service` (`RestClient`, без таймауту);
3. після відповіді статус оновлюється на `PAID` (коротка транзакція).

> **Проєктне рішення.** HTTP-виклик стороннього сервісу (і так само імітація
> провайдера в payment) свідомо винесені **за межі транзакції БД**. Утримувати
> з'єднання HikariCP протягом мережевого виклику/затримки — антипатерн: пул
> вичерпується першим, і вузьким місцем стає БД, а не потоки. Винісши виклики
> назовні, ми робимо єдиним binding-обмеженням baseline саме **пул потоків
> Tomcat (50)**. Платою є втрата атомарності між замовленням і платежем — її
> розв'язує патерн **Transactional Outbox + Saga** у Розділі 3.4.

### Свідомі дефекти baseline (точка відліку)

| Дефект | Наслідок |
|---|---|
| Платформні потоки Tomcat (50) | Кожен запит утримує потік на весь час оплати → стеля ≈ 250 RPS |
| Блокуючий виклик **без таймауту** | Потік не звільняється, поки payment-service не відповість |
| Відсутність механізмів стійкості | Немає ізоляції, відсічення, деградації (Bulkhead/CB/Timeout) — Розділ 3.2 |
| Немає атомарності замовлення↔платіж | За збою payment замовлення «зависає» в `PENDING` → мотивація Outbox/Saga (3.4) |

### Теоретичне обґрунтування (закон Літтла)

Необхідна паралельність = RPS × час обробки. За затримки оплати 200 мс пул із
**50 потоків** здатен утримувати щонайбільше `50 / 0.2с = 250` одночасних
запитів на стадії оплати. Це і є теоретична стеля пропускної здатності baseline:
будь-яке навантаження понад ≈ 250 RPS переповнює пул і чергу, час відповіді
зростає лавиноподібно, а надлишок отримує `503`/timeout.

### Тест 1

Подаємо навантаження до 1000 RPS (`load-tests/baseline.js`).

**Очікуваний результат:** насичення пулу потоків Tomcat, зростання черги, час
відповіді сягає секунд, частина клієнтів отримує таймаути / `503`. SLO порушено.

### Результати Тесту 1

Профіль: ramp до 1000 RPS, утримання, спад (тривалість 2 хв 40 с). Звіт —
`load-tests/summary.html`.

| Показник | Значення |
|---|---|
| Цільове навантаження (пік) | 1000 RPS |
| Пропускна здатність (оброблено, факт.) | **251.7 запит/с** |
| Успішних (`201`) | 25 905 → **≈ 162 замовл./с** |
| **Частка помилок** | **35.7 %** (`dial: i/o timeout` — відмова з'єднання) |
| Запитів відкинуто клієнтом (`dropped`) | 43 439 |
| p95 латентності (обслужені) | 625 мс |
| max латентності | 60 с |
| CPU order у піку | ~0.9 ядра з 2 (не насичено) |
| SLO (помилки < 1%) | ❌ (35.7 %) |

**Висновок.** Пропускна здатність зафіксувалась на рівні **251.7 запит/с** —
тотожно теоретичній стелі `50 потоків / 0.2 с = 250 RPS` (закон Літтла). Це
**прямий доказ**, що binding-обмеженням є саме пул платформних потоків Tomcat:
CPU при цьому завантажений лише на ~45 % (0.9 з 2 ядер), тобто **відмова
архітектурна, а не апаратна** — додавання ядер її не усуває. Через обмежену
чергу (`max-connections=150`) надлишок понад стелю не накопичується нескінченно,
а **відсікається** (35.7 % запитів отримують відмову з'єднання). Система не
здатна обслужити цільові 1000 RPS і втрачає понад третину трафіку. Це і є точка
відліку, відносно якої вимірюються всі наступні архітектурні покращення.

### Гілка
`master` — базова архітектура (точка відліку). Кожне наступне покращення —
окрема гілка, відмерджена від попередньої.

## 3.2. Усунення ресурсних вузьких місць

Три послідовні кроки, кожен змінює **рівно одну** річ за незмінного стенду. Звіти
по кожному кроці — у `load-tests/reports/`.

### 3.2.1. Віртуальні потоки

**Що змінено:** `spring.threads.virtual.enabled=true` на обох сервісах (єдина
змінна). Tomcat обслуговує кожен запит окремим **віртуальним** потоком; пул із 50
платформних потоків більше не є стелею, а блокування на виклику payment коштує
дешево (віртуальний потік від'єднується від носійного на час очікування).

**Результати** (той самий `baseline.js`, профіль до 1000 RPS):

| Показник | Baseline (3.1) | Віртуальні потоки (3.2.1) | Зміна |
|---|---|---|---|
| Успішних замовлень/с | 162 | **370** | **×2.3** |
| Пропускна здатність (всього) | 251.7/с | 445.3/с | ×1.8 |
| Частка помилок | 35.7 % | **16.9 %** | ↓ удвічі |
| p50 латентності | 605 мс | **203 мс** | ≈ час payment |
| p95 латентності | 625 мс | 323 мс | ↓ майже вдвічі |
| CPU order у піку | ~0.9 / 2 ядра | **~2.0 / 2 ядра** | насичено |

**Висновок (вирішено частково).** Прибравши штучний ліміт у 50 потоків,
віртуальні потоки підняли успішну пропускну здатність у **2.3 раза** і **вдвічі**
зменшили частку помилок; медіана впала до 203 мс — це чистий час обробки payment,
тобто черга практично зникла. Проте проблему **не розв'язано повністю**:

1. **Вузьке місце переїхало на CPU** — order насичує обидва ядра (200 %). За
   синхронної блокуючої моделі кожен запит дорогий (HTTP-клієнт, серіалізація,
   GC при тисячах одночасних потоків). Межу заліза прибере лише неблокуюча
   модель (async/Kafka, Розділ 3.4).
2. **Немає відмовостійкості.** Необмежена конкурентність означає, що при
   деградації payment (повільні відповіді чи відмова) order безконтрольно
   плодитиме віртуальні потоки в очікуванні, без швидкої відмови (fail-fast) та
   ізоляції ресурсів. Систему все ще можна «завалити» через залежність.

Саме проблема №2 веде до наступного кроку.

**Технічна примітка (pinning).** Класичне обмеження віртуальних потоків —
*pinning*: віртуальний потік «пришпилює» носійний і не звільняє його, якщо
блокування відбувається всередині `synchronized` або нативного коду. До JDK 21–23
це знецінювало віртуальні потоки на ділянках із `synchronized` (зокрема в деяких
драйверах БД). У нашому стенді (**Java 25**) діє **JEP 491** — `synchronized`
більше **не спричиняє pinning**, тож блокування на виклику payment і на JDBC
коректно від'єднують носійний потік. Це підтверджує, що отриманий приріст —
реальний, а не замаскований pinning'ом (за потреби перевіряється подіями JFR
`jdk.VirtualThreadPinned` або `-Djdk.tracePinnedThreads=full`).

**Гілка:** `virtual-threads` (від `master`).

### 3.2.1.1. Демонстрація проблеми: деградація payment

> **Уявімо, що затримка на сервері payment зросла** (перевантажений зовнішній
> провайдер, проблеми з мережею тощо) — і запити почали відвалюватись.

Щоб показати наочно, ми виконали **ін'єкцію відмови** (fault injection): підняли
затримку payment з 200 мс до **2000 мс** (`PAYMENT_DELAY_MS=2000`), не змінюючи
нічого іншого, і подали **однакове помірне навантаження 300 RPS** (сценарій
`load-tests/fault-injection.js`) у двох станах. Віртуальні потоки — увімкнені
(тобто це наша поточна найкраща версія).

| Показник | payment 200 мс (норма) | payment 2000 мс (деградація) |
|---|---|---|
| Успішних замовлень/с | **202** | **45** |
| Частка помилок | 4.4 % | **63.6 %** |
| p95 латентності | 206 мс | **2.01 с** |
| CPU order | у нормі | ~30 % (потоки **заблоковані**, не рахують) |

**Звіти:** `reports/3.2.1-payment-healthy.html` та `reports/3.2.1-payment-slow.html`.

**Діагноз.** Десятикратне сповільнення payment **обвалює** order: успішна
пропускна здатність падає з 202 до 45 замовл./с, частка помилок зростає до
**63.6 %**. Причина — у order **немає жодного захисту**: він продовжує приймати
запити й синхронно блокуватись на повільному payment, аж поки не вичерпає ліміт
з'єднань (стеля `150 / 2 с = 75` обробок/с), після чого решта отримує відмову.
CPU при цьому низький — ресурси не «згоряють», вони **простоюють заблокованими**.

**Висновок.** Віртуальні потоки підняли пропускну здатність, але **не дають
відмовостійкості**: повільна або несправна залежність каскадно тягне сервіс за
собою. Потрібен механізм, який **виявить деградацію payment і швидко відмовить**
(fail-fast), не даючи запитам накопичуватись. Це й розв'язує наступний крок.

### 3.2.2. Circuit Breaker + Timeout

**Що змінено.** Навколо виклику payment додано два механізми (Resilience4j core,
програмно — без Spring Boot-стартера, заради сумісності з Boot 4):

- **Timeout 500 мс** на HTTP-виклик (`HttpClientSettings`). payment гальмує, а не
  падає; таймаут перетворює повільну відповідь (2 с) на помилку за 500 мс.
- **Circuit Breaker** на `PaymentClient.charge`: ковзне вікно 20 викликів,
  поріг невдач 50 %, повільний виклик > 400 мс, відкрите кільце тримається 5 с,
  потім 5 пробних викликів (half-open). При розмиканні чи таймауті — кидаємо
  `PaymentUnavailableException`, контролер віддає **`503` миттєво** (fail-fast).

**Умови:** той самий fault-injection — `payment=2000мс`, 300 RPS, vthreads увімкнені.

| Показник | Без CB (3.2.1-payment-slow) | **З CB (3.2.2)** |
|---|---|---|
| Тип відмови | `status 0` — connection timeout | **`503`** — розумна відмова |
| Запитів оброблено сервером | 9 430 (≈123/с; решта не достукалась) | **13 501 (≈295/с — усі 300 RPS)** |
| Успішних `201` | 45/с | 0/с |
| Відмов | 6 001 conn-timeout (63.6 %) | 13 501 × `503` (100 %, швидкі) |
| p50 латентності | повільно | **1.48 мс** |
| p95 латентності | **2.01 с** | 604 мс |
| Накопичення (max VUs) | 4 455 | **276** |
| Стан order | захлинувся, з'єднання вичерпані | **живий, чуйний, ресурси вільні** |

**Звіти:** `reports/3.2.2-circuit-breaker.html` vs `reports/3.2.1-payment-slow.html`.

**Два типи відмов (суть розділу).** Без CB запити «відвалюються», бо сервер
**захлинувся**: кожен висить ~2 с і вичерпує з'єднання (`status 0`). З CB кожен
запит одразу отримує **`503` за ~1.5 мс** — це **«розумна» відмова за рішенням
Circuit Breaker**, а не колапс. Сервер встиг відповісти на **всі** 300 RPS, без
накопичення (276 VU проти 4 455).

**Життєвий цикл кільця (з логів `order-service`).** Підтверджено керовану
деградацію та **самовідновлення**:
```
CLOSED → OPEN          payment деградував, поріг невдач перевищено
OPEN  ↔ HALF_OPEN      кожні 5 с проба; payment досі повільний → знову OPEN
HALF_OPEN → CLOSED     payment ожив (200 мс) → кільце замкнулось, трафік відновлено
```

**Висновок (чесний нюанс).** Circuit Breaker **не повертає** успішні замовлення,
поки payment реально недоступний (0 успішних: 2 с > таймаут 500 мс) — і не
повинен. Його роль — **локалізувати відмову**: замість каскадного обвалу система
**деградує керовано** — швидкий `503`, сервер живий, ресурси вільні, клієнт
миттєво отримує зворотний зв'язок, а щойно payment одужує — кільце замикається
**саме**. Ті 45 успішних/с без CB коштували 2 с латентності й забитого сервера —
сумнівна цінність. Це і є визначення відмовостійкості: **деградувати, але не
падати**.

### 3.2.2.1. Часова шкала «outage + recovery» (Grafana)

Щоб цінність CB було видно наочно, той самий сценарій прогнали **двічі — з CB і
без CB** (перемикач `PAYMENT_RESILIENCE_ENABLED`), з ін'єкцією відмови в рантаймі
через chaos-ендпоінт `POST /admin/delay`. Метрики транслюються
**k6 → Prometheus → Grafana** (дашборд «Circuit Breaker — Outage & Recovery
Timeline», `http://localhost:3000`).

Профіль: стале навантаження 150 RPS; **0–30с** payment здоровий (200 мс) →
**30–65с** деградація (2 с) → **65–100с** відновлення (200 мс).

Показники у **фазі деградації** (надійні середні через `increase()` за вікно):

| Метрика (payment = 2 с) | **CB OFF** | **CB ON** |
|---|---|---|
| Успішних (повільних, ~2 с) /с | ~68 | 0 |
| Швидких `503` (fail-fast) /с | 0 | **~145** |
| Conn-timeout — сервер захлинувся /с | **~26** | **0** |
| Хвіст латентності | **десятки секунд** | суб-секунда |
| Відновлення після оживання payment | з лагом/беклогом | **миттєве** (CB сам замикається) |

**Звіти:** `reports/3.2.2-timeline-cb-on.html` та `reports/3.2.2-timeline-cb-off.html`;
живі графіки — Grafana.

**Як читати дашборд (зверху вниз, синхронно по часу):**

| Панель | Маркер | Сенс |
|---|---|---|
| ① Затримка payment | сходинка 200 → 2000 мс | момент ін'єкції відмови |
| ② Стан Circuit Breaker | червона лінія `OPEN` → 1 | **момент спрацювання патерну** (далі OPEN↔HALF_OPEN кожні 5 с; `CLOSED`→1 = авто-відновлення) |
| ③ Потік за типом | `успішно 201`↓, `503 fail-fast`↑ | сервер відмовляє швидко, не висить |
| ④/⑤ латентність, % помилок | сплеск | наслідок деградації |

Стан CB експортується метрикою `resilience4j_circuitbreaker_state`, затримка —
`payment_processing_delay_ms` (обидві скрейпляться Prometheus з `/actuator/prometheus`).
У прогоні **без CB** панель ② лишається `CLOSED` (кільце вимкнене), а в ③ замість
`503` росте `conn timeout` — наочний контраст «розумна відмова vs обвал».

> **Чесний нюанс (важливо для захисту).** За **часткової** деградації (payment
> повільний, але живий) CB **не збільшує** кількість успішних замовлень — він
> навіть зменшує її, бо відрубує запити, які без CB завершились би (повільно, за
> 2 с). Цінність CB не в пропускній здатності, а в **обмеженій латентності**,
> **здоров'ї сервера** (0 дропів проти ~26/с), **миттєвому відновленні** та
> захисті від повної відмови й каскаду. Тобто плата за стійкість — відмова від
> неприйнятно повільних «успіхів».

**Чого CB не вирішує (місток до Bulkhead).** Circuit Breaker захищає від хворого
downstream загалом, але **не ізолює** конкурентний доступ до спільного ресурсу
(напр. пулу БД) між різними операціями: одна повільна операція все ще може з'їсти
весь пул і «затягнути» інші. Це розв'язує наступний крок.

**Гілка:** `circuit-breaker` (від `virtual-threads`).

### 3.2.3. Bulkhead — ізоляція пулу з'єднань БД

**Що змінено.** Доступ до БД (метод збереження/оновлення замовлення) огорнуто
**семафорним Bulkhead** (Resilience4j) із лімітом **8 одночасних звернень — меншим
за розмір пулу HikariCP (10)**. Ідея: пул ніколи не вичерпується повністю, бо до
нього не може звернутися більше операцій, ніж дозволяє семафор; надлишок одразу
отримує `503` (fail-fast), а не стає в чергу за вільним з'єднанням.

Circuit Breaker (3.2.2) захищає від хворого **downstream** (payment), але не
ізолює **спільний ресурс** — пул з'єднань. Якщо під навантаженням запити до БД
сповільнюються, з'єднання утримуються довше, і за необмеженої конкурентності
(віртуальні потоки) пул вичерпується. Саме це й розв'язує Bulkhead.

**Методика.** Щоб відтворити контенцію пулу, додано імітацію повільного запиту
(`ORDER_DB_WORK_DELAY_MS=100` — транзакція утримує з'єднання 100 мс). Стале
навантаження 100 RPS, 60 с; прогін двічі — з bulkhead і без (`ORDER_BULKHEAD_ENABLED`).

| Показник | **Без Bulkhead** | **З Bulkhead (ліміт 8)** |
|---|---|---|
| HikariCP active (макс.) | **10 — пул вичерпано** | **8 — обмежено** |
| HikariCP **pending** (макс.) | **128 потоків у черзі** | **0** |
| Час отримання з'єднання (макс.) | **~5 с** | **62 мс** |
| Семафор (вільні дозволи, мін.) | — (не задіяний) | **0** (насичений, відсікає) |
| p95 латентності | **5.77 с** | низька (швидкий `503`) |
| Тип відмови | `conn-timeout`/`500` (повільно) | `503` fail-fast |
| Успішних замовлень/с | ≈ 28 | ≈ 17 |

**Звіти/дашборд:** `reports/3.2.3-bulkhead-{on,off}.*`; окремий дашборд Grafana
«Bulkhead — Ізоляція пулу з'єднань БД» (панелі: active/pending пулу, вільні дозволи
семафора, час отримання з'єднання, потік за типом).

**Висновок.** Без ізоляції 129 потоків стоять у черзі за 10 з'єднаннями, час
очікування сягає 6.5 с, латентність — секунди; пул фактично вичерпано. Bulkhead
утримує одночасний доступ нижче ємності пулу: active ≤ 8, черга порожня (0),
з'єднання видається за мілісекунди, а надлишок відсікається миттєво. Як і Circuit
Breaker, Bulkhead **не підвищує пропускну здатність** (вона навіть трохи нижча —
плата за обмеження конкурентності), а **гарантує, що спільний ресурс не
вичерпається**, а перевантаження виявляється як швидка керована відмова замість
каскадного зростання черги й латентності.

**Гілка:** `bulkhead` (від `circuit-breaker`).
